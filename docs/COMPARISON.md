# Aftercare vs Superpowers

## Shared DNA

Aftercare intentionally borrows the strongest ideas from the Superpowers style of agentic development:

- skill-first composition
- short, trigger-rich descriptions
- explicit workflows instead of vague suggestions
- reusable references rather than giant monolithic instructions
- host-native adoption across multiple coding agents

## The difference in motivation

Superpowers is primarily about **how to build software with coding agents from the beginning**.
Aftercare is about **how to recover, discipline, and harden software once implementation already exists**.

## Different center of gravity

### Superpowers asks
- What are we building?
- What is the spec?
- What is the implementation plan?
- How do we execute it task by task?

### Aftercare asks
- What did the implementation do to the repo?
- Where did boundaries drift?
- What refactor is actually worth doing now?
- What must be fixed before merge and what can wait?

## Why this distinction matters

Most teams already have strong generation workflows.
Their problem is the growing pile of code that technically works but gets structurally worse every week.

Aftercare is intentionally narrower and more judgment-heavy.
It is not trying to replace an end-to-end methodology.
It is trying to become the default **post-dev governance layer**.
