# Project Structure Review — Anti-Patterns

Start with the global list in [`../../references/anti-patterns.md`](../../references/anti-patterns.md). The entries below are specific to project structure and extend it.

## Framework-shaped top level

The repo's top level mirrors a framework's idioms (`controllers/`, `services/`, `models/`) instead of the product's domain. Engineers must mentally translate from product terms to framework terms on every navigation. Often the result of scaffolding generators never being customized.

**Severity**: Medium in small repos, High when the domain is complex.
**Fix**: introduce product-shaped top levels (`billing/`, `search/`, `onboarding/`) and let framework shapes live inside them.

## Phantom modules

A directory exists for a capability the product once considered but never built, or that was deprecated without removal. Each new engineer loses time understanding why it is there.

**Severity**: Low individually, Medium when several accumulate.
**Fix**: delete empty and near-empty directories. If you are saving a name for later, do it in a design doc, not the file tree.

## One-feature shared

`shared/`, `common/`, `utils/`, or `lib/` exists but 60%+ of its contents are used by exactly one feature. The shared location is doing private work in a public place.

**Severity**: Medium.
**Fix**: move the one-feature code back into the feature. Reserve `shared/` for code that genuinely crosses features.

## Depth for its own sake

Deeply nested folders (`src/app/modules/features/billing/components/invoice/...`) create long import paths without reflecting real structure. Engineers paste long relative imports or add alias tricks.

**Severity**: Medium.
**Fix**: flatten. Target depth is the minimum that separates the axes the repo actually has (feature × layer, or feature × runtime). Three levels below `src/` is usually enough.

## Scaffolded-but-unowned

A directory was generated by a tool or template and has accumulated drift, but no one remembers why it is split the way it is. New code is forced into shapes that no longer fit.

**Severity**: Medium.
**Fix**: document the intended shape in a README inside the directory, or restructure it. "Unowned" is the root cause; fix that first.

## Scattered feature

One feature's code is split across three or four top-level directories without a clear rule. Changing the feature requires touching each location and remembering which piece lives where.

**Severity**: High when the feature changes frequently.
**Fix**: cluster. The feature should have a home directory, and integrations into other layers should be named (e.g., `billing/http/`, `billing/db/`) rather than scattered.

## Tests-far-from-code

Tests live in a central `tests/` directory mirroring the source tree — or worse, mirroring an older shape of the source tree. When code moves, tests do not, and the mirror drifts.

**Severity**: Medium; often a leading indicator of the test-illusion anti-pattern from [`../../references/anti-patterns.md`](../../references/anti-patterns.md).
**Fix**: colocate tests with the code they test, or establish a hard rule that the `tests/` mirror is updated in the same PR as source moves.

## Private-in-public naming

A directory named generically (`core/`, `internal/`, `utils/`) contains domain-specific code that should be named for the domain. Readers pay attention to `billing.ts` and ignore `core.ts`, even when `core.ts` contains more business logic.

**Severity**: Medium.
**Fix**: rename. Generic names are a promise the code rarely keeps.
